# 模块化机器人栈开发规划（中文）

## 1. 文档目的

本文档用于整理当前阶段已经讨论并达成共识的开发内容，明确：

- 当前主线模块有哪些
- 还缺哪些系统骨架层
- 各部分建议优先级如何安排
- 后续代码应分别落到仓库中的哪个模块边界

本文档是开发规划，不代表这些能力已经在当前仓库中全部实现。

## 2. 当前架构共识

本项目坚持模块化路线，不走“自然语言直接出动作”的黑盒路径。当前讨论后形成的目标链路如下：

`自然语言 -> 任务解析 -> 任务中间表示 -> 语义绑定 -> 世界状态 -> 高层任务规划 -> 技能执行 -> 几何/轨迹规划 -> 底层闭环控制 -> 评估与回放`

对 `PDDL` 与 `STL` 的定位也已经明确：

- `PDDL`：用于高层离散任务规划，负责动作顺序、前置条件、后置条件与目标可达性。
- `STL`：用于执行阶段的时序与安全约束监视，负责持续检查“执行过程是否违规”。
- 技能层、运动规划层、控制层：才是真正的执行层。

因此，不应把 `PDDL` 或 `STL` 设计成“最终万能执行器”。更合理的设计是：

- `PDDL` 负责“先做什么、后做什么”
- `STL` 负责“执行过程中什么必须始终满足”
- 控制器负责“机器人具体怎么动”

## 3. 已确认的主线开发项

下表整理了当前已明确提出的开发内容。

| 编号 | 模块 | 作用 | 优先级 |
| --- | --- | --- | --- |
| A | 自然语言到任务中间表示 | 将自然语言指令解析成结构化任务表示，而不是直接输出控制命令 | P0 |
| B | 点云重建网络 | 从视觉输入恢复可用于后续几何建模的三维表示 | P0 |
| B.1 | 场景图构建 | 构建包含物体类别、六自由度位姿、空间关系的场景图 | P1 |
| AB | `PDDL` / `STL` | `PDDL` 用于高层规划，`STL` 用于时序与安全约束监视 | P2 |
| C | 运动基元 | 提供基础运动能力的可复用原语 | P1 |
| C.1 | 操作技能 | 提供抓取、放置、递交等组合式操作能力 | P2 |
| C.2 | 搜索与优化 | 包括 `RRT*`、`CHOMP` 以及轨迹层的优化方法 | P3 |
| D | 闭环控制 | 将高层计划落到可执行的连续控制流程 | P0 |
| D.1 | 实时状态估计 | 通过编码器、力觉、视觉等信息更新机器人与环境状态 | P0 |
| D.2 | 模型预测控制 | 在控制层进行约束优化与连续决策 | P0 |
| D.3 | 分层强化学习 | 作为高层符号与低层连续之间的可选增强方向 | 远期方向 |

根据当前讨论，以下两项不再作为“当前遗漏项”统计：

- 机器人本体模型
- 仿真与硬件适配层

这两部分已由现有工作覆盖，后续只需在接口上与新模块保持一致。

## 4. 当前最关键的遗漏部分

虽然主线模块已经列出，但系统要真正闭环，还缺少几层关键骨架。如果不补，这些模块会变成并列堆叠，而不是可运行系统。

### 4.1 统一任务中间表示

必须定义一份稳定、可验证、可追踪的任务中间表示，用来承接“自然语言理解”和“高层规划”之间的接口。

这份表示至少应包含：

- 任务标识
- 动作意图
- 目标对象
- 目标位置或目标对象
- 前置条件
- 后置条件
- 约束
- 失败恢复策略
- 可追踪的子步骤标识

这部分应优先落在 `interfaces/` 与 `modules/task_parser/`。

### 4.2 世界状态存储层

必须有一个统一的世界状态层，用于保存：

- 场景中的对象列表
- 每个对象的类别、位姿与关系
- 机器人自身状态
- 当前任务执行后的状态更新
- 置信度、来源与历史快照

这部分应由 `modules/world_model/` 统一维护，而不是分散在感知、规划、控制各处。

### 4.3 语义绑定层

自然语言中的对象名称、代词和关系表达，必须绑定到场景中的具体实例。

例如：

- “杯子”对应哪个对象实例
- “递给我”中的“我”对应哪个人体目标区域
- “桌子左边那个红色瓶子”对应哪个场景节点

这部分应放在 `modules/grounding/`，而不是放在规划器或控制器中临时处理。

### 4.4 执行管理与重规划层

系统还需要一个负责执行过程调度的管理层，用来统一处理：

- 当前该执行哪个子步骤
- 某一步前置条件是否满足
- 技能执行失败后是重试、回退还是重规划
- 执行过程中如何记录结构化轨迹
- 高层任务状态如何更新

这部分是高层规划和低层执行之间的桥梁，应明确放在 `modules/planner/` 与 `modules/control/` 的边界上。

### 4.5 技能描述规范

仅有“技能库”还不够，还需要为每个技能定义清晰的描述规范，至少包括：

- 技能名称
- 输入
- 输出
- 前置条件
- 成功条件
- 失败码
- 所需几何条件
- 所需控制模式

这部分建议放在 `interfaces/` 与 `modules/skills/` 中统一维护。

### 4.6 几何可行性检查层

高层任务逻辑成立，不代表几何上就能执行。还需要补一层专门做：

- 可达性检查
- 抓取姿态可行性检查
- 放置位姿可行性检查
- 碰撞检查
- 轨迹存在性检查

这层通常位于技能层与运动规划层之间。

### 4.7 感知前端

点云重建网络前面还需要配套的感知前端能力，包括：

- 传感器采集
- 标定
- 时间同步
- 多视角对齐
- 目标分割或检测
- 目标跟踪

否则重建网络缺乏稳定、统一的输入来源。

### 4.8 评估、轨迹记录与回放

本项目的目标不是演示式成功，而是可审计、可调试、可复现，因此还必须补齐：

- 标准化任务集
- 成功与失败分类
- 结构化执行轨迹
- 回放机制
- 消融与对比评估

这部分应由 `eval/` 与结构化 `trace` 共同承担。

### 4.9 运行时安全监视

即使后续引入 `STL`，仍需要将运行时安全机制明确为独立职责，包括：

- 碰撞保护
- 力阈值保护
- 工作空间边界保护
- 超时保护
- 急停与看门狗机制

### 4.10 不确定性处理

机器人系统必须面对不确定输入，因此需要为以下问题预留表示与处理机制：

- 语言解析歧义
- 目标识别置信度不足
- 位姿估计误差
- 感知暂时丢失目标
- 执行后世界状态与预期不一致

如果没有这一层，系统会默认世界状态完全准确，后续规划与控制都容易出问题。

## 5. 推荐的优先级重排

为了尽快形成可运行、可验证的闭环，建议将开发重点按以下顺序组织。

### P0：最小可闭环系统

P0 目标是先形成“能跑通且能定位错误”的最小系统，不追求一次性做大做全。

建议放入 P0 的部分：

- 自然语言到任务中间表示
- 世界状态存储层
- 语义绑定层
- 执行管理与重规划基础框架
- 实时状态估计
- 闭环控制
- 控制层 `MPC`
- 点云重建基础能力
- 结构化轨迹记录
- 安全监视基础能力

### P1：补齐几何与可解释能力

P1 重点是让系统不只是“能跑”，还要具备较强的可解释几何基础。

建议放入 P1 的部分：

- 场景图构建
- 运动基元
- 几何可行性检查
- 感知前端工程化能力
- 任务回放与诊断工具

### P2：加入更强的符号规划与时序约束

P2 可以在基础闭环稳定后再补，不应反客为主成为第一阶段阻塞点。

建议放入 P2 的部分：

- `PDDL` 高层任务规划
- `STL` 时序与安全约束表达
- 操作技能库扩展
- 更丰富的失败恢复策略
- 不确定性建模的进一步完善

### P3：研究型增强能力

这些方向有价值，但不应阻塞当前主线。

建议放入 P3 的部分：

- `RRT*`
- `CHOMP`
- 更复杂的轨迹优化器
- 分层强化学习

## 6. 推荐的模块归属

为了符合当前仓库的模块边界，建议按以下方式分配职责。

### `interfaces/`

负责定义跨模块共享的结构化接口，包括：

- 任务中间表示
- 技能描述规范
- 世界状态结构
- 执行轨迹结构
- 失败码与恢复策略结构

### `modules/task_parser/`

负责将自然语言解析成结构化任务表示，不直接处理几何和控制问题。

### `modules/grounding/`

负责将任务中的语义目标绑定到场景中的具体对象、区域与候选位姿。

### `modules/world_model/`

负责存储、更新、查询世界状态，并维护状态快照和来源信息。

### `modules/planner/`

负责高层任务分解、步骤排序、前置条件检查、重规划触发与任务级执行管理。

### `modules/skills/`

负责维护运动基元、操作技能及其结构化描述。

### `modules/control/`

负责底层闭环控制、连续执行、力位姿控制与状态反馈。

### `eval/`

负责基准任务、结构化报告、回放、消融与评估协议。

## 7. 当前阶段的设计约束

以下约束在当前讨论中已经形成共识，后续设计应尽量遵守：

- 不让 `LLM` 直接无约束地产生最终 `PDDL`
- 不跳过世界状态层，直接从语言连到控制
- 不把 `PDDL` 误当成控制执行器
- 不把 `STL` 误当成高层任务分解器
- 不把感知、规划、控制写成一个不可追踪的大模块
- 每一层都应能输出可审计的结构化中间结果

## 8. 后续建议

如果继续往下推进，建议优先完成以下三份更细的设计文档：

1. 任务中间表示设计文档
2. 世界状态与语义绑定设计文档
3. 执行管理、重规划与轨迹记录设计文档

这三份文档一旦稳定，后续再接 `PDDL`、`STL`、技能库与控制器，会明显更稳。
